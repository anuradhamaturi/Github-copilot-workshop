# Task Manager CLI Technical Design

## 1. Data models

### Task

| Property | Type | Required | Validation rules |
| --- | --- | --- | --- |
| id | string | Yes | Must be a non-empty string; generated by the app at create time; immutable after creation. |
| title | string | Yes | Trimmed value must be non-empty; maximum length 120 characters. |
| description | string | Yes | Defaults to empty string when omitted; maximum length 1000 characters. |
| status | 'todo' \| 'in-progress' \| 'done' | Yes | Must be one of the allowed status values; defaults to 'todo' on create. |
| priority | 'low' \| 'medium' \| 'high' | Yes | Must be one of the allowed priority values; defaults to 'medium' on create. |
| createdAt | string | Yes | Must be an ISO-8601 timestamp string; set automatically on create; immutable after creation. |
| updatedAt | string | Yes | Must be an ISO-8601 timestamp string; set on create and updated after successful updates. |

Notes:
- No database model is used; task records are kept only in memory for the process lifetime.

Detailed property validation rules:
- id
  - Type must be string.
  - Value must be non-empty after trim.
  - Generated internally only during create.
  - Must be unique within TaskStore.tasks.
  - Cannot be changed by update operations.
- title
  - Type must be string.
  - Value is normalized with trim().
  - Minimum length after trim: 1.
  - Maximum length: 120.
  - Required on create.
- description
  - Type must be string.
  - Defaults to empty string if omitted on create.
  - Maximum length: 1000.
  - Leading and trailing whitespace may be trimmed before storage.
- status
  - Type must be string.
  - Allowed values: todo, in-progress, done.
  - Defaults to todo if omitted on create.
  - Invalid enum values are rejected.
- priority
  - Type must be string.
  - Allowed values: low, medium, high.
  - Defaults to medium if omitted on create.
  - Invalid enum values are rejected.
- createdAt
  - Type must be string.
  - Must be a valid ISO-8601 UTC timestamp.
  - Set once during create.
  - Cannot be changed by update operations.
- updatedAt
  - Type must be string.
  - Must be a valid ISO-8601 UTC timestamp.
  - Set during create.
  - Must be refreshed only after a successful update.
  - Must not change when validation fails or no fields are changed.

### TaskStore

| Property | Type | Required | Validation rules |
| --- | --- | --- | --- |
| tasks | Task[] | Yes | In-memory array of Task objects; initialized as an empty array at startup. |

## 2. File structure

```text
src/
  index.js                    # CLI entry point, argument parsing, command dispatch, top-level error boundary
  constants/
    enums.js                  # Allowed status/priority values and priority ranking map
  models/
    task.js                   # Task creation helpers and task field validation
  services/
    task-service.js           # CRUD operations plus filtering and sorting against in-memory store
  utils/
    errors.js                 # Error classes/factories and user-facing error mapping
    format.js                 # stdout formatting for list and command result output
```

## 3. Module responsibilities

### src/index.js
- Exports:
  - run(argv): Promise<void>
- Responsibilities:
  - Parse CLI commands and flags for create, list, update, delete.
  - Call service-layer methods with validated command payloads.
  - Print success output to stdout and errors to stderr.
  - Set process.exitCode = 1 for command/validation failures.
- Depends on:
  - src/services/task-service.js
  - src/utils/format.js
  - src/utils/errors.js
  - src/constants/enums.js

### src/constants/enums.js
- Exports:
  - STATUS_VALUES
  - PRIORITY_VALUES
  - PRIORITY_RANK
- Responsibilities:
  - Centralize allowed enum values and priority sort precedence.
- Depends on:
  - No internal modules.

### src/models/task.js
- Exports:
  - createTask(input)
  - validateCreateInput(input)
  - validateUpdateInput(input)
- Responsibilities:
  - Enforce task field constraints.
  - Apply defaults for status, priority, and description.
  - Build normalized Task objects with timestamps.
- Depends on:
  - src/constants/enums.js
  - src/utils/errors.js

### src/services/task-service.js
- Exports:
  - createTaskRecord(input)
  - listTaskRecords(options)
  - updateTaskRecord(id, patch)
  - deleteTaskRecord(id)
- Responsibilities:
  - Maintain in-memory TaskStore tasks array.
  - Implement create/list/update/delete behavior.
  - Apply list filters (status, priority).
  - Apply sorting (priority, createdAt, order asc/desc).
- Depends on:
  - src/models/task.js
  - src/constants/enums.js
  - src/utils/errors.js

### src/utils/errors.js
- Exports:
  - ValidationError
  - NotFoundError
  - UsageError
  - toUserMessage(error)
- Responsibilities:
  - Define stable error categories.
  - Standardize messages and map unknown/internal errors to safe user-facing text.
- Depends on:
  - No internal modules.

### src/utils/format.js
- Exports:
  - formatTask(task)
  - formatTaskList(tasks)
  - formatSuccess(message)
- Responsibilities:
  - Render consistent CLI output for single-task and list responses.
- Depends on:
  - No internal modules.

Dependency direction:
- index -> services -> models/constants/utils
- models -> constants/utils
- constants and utils do not depend on services or index

## 4. Error handling strategy

Error types:
- VALIDATION_ERROR
- NOT_FOUND_ERROR
- USAGE_ERROR

Where errors are thrown:
- VALIDATION_ERROR
  - src/models/task.js for invalid title, description, status, priority, immutable field edits, and empty update payloads.
  - src/index.js for invalid flag values (sort/order/filter) or missing required command arguments.
- NOT_FOUND_ERROR
  - src/services/task-service.js when update/delete targets an id that is not present in TaskStore.
- USAGE_ERROR
  - src/index.js for unknown commands, unsupported flags, and malformed command syntax.

Handling behavior:
- src/index.js catches thrown errors at the command boundary and prints Error: <message> to stderr.
- src/index.js sets process.exitCode = 1 for handled failures.
- Unknown internal errors are converted to a generic user-facing message by src/utils/errors.js and logged with console.error.
- Success output is written to stdout only.
